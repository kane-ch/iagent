# KB Answer Format

## 基本原则

基于财报知识库回答时，答案必须遵循：

1. 先检索，后回答。
2. 只基于检索结果作答。
3. 明确区分事实、引用和推断。
4. 不编造知识库中没有的信息。
5. 尽量给出可追溯 citation。

## 推荐回答结构

### 1. 简短结论

用 1–3 句话回答用户核心问题。

示例：

> 根据知识库中检索到的财报内容，经营利润下降主要与成本上升、销售/研发费用增加以及产品结构变化有关。

### 2. 分点说明原因

每个原因用一条 bullet，包含：

- 原因标题。
- 证据摘要。
- 引用信息。

示例：

```markdown
- **成本压力增加**：财报提到成本上升与原材料、制造和履约相关支出增加有关。  
  来源：LI, 20-F, 2026-04-10, Management Discussion, document_id=...
```

### 3. 汇总判断

把多个原因合并成整体判断，但不要超出证据范围。

示例：

> 综合来看，利润下降不是单一因素造成，而是收入增长放缓、成本端压力和经营费用投入共同作用的结果。

### 4. 不确定性说明

如果证据不足，应明确说明。

示例：

> 当前知识库中未检索到管理层对该指标下降的直接解释。以上结论仅基于相关成本和费用段落的归纳，建议补充构建最新财报知识库后再次检索。

## 标准输出模板

```markdown
### 结论
<一句话总结>

### 主要原因
1. **<原因1>**：<证据摘要>  
   来源：<ticker> <form_type> <filing_date> <section/source> <document_id>

2. **<原因2>**：<证据摘要>  
   来源：<ticker> <form_type> <filing_date> <section/source> <document_id>

### 说明
<如有缺失、限制、推断或需要进一步检索，在这里说明。>
```

## 引用字段使用

`retrieve_filing_kb` 返回 chunk 时可能包含：

- `ticker`
- `documentId`
- `formType`
- `fiscalYear`
- `fiscalPeriod`
- `filingDate`
- `sourceFileName`
- `sectionTitle`
- `citation`
- `score`

优先引用：

```text
ticker + formType + fiscalYear/fiscalPeriod + filingDate + sectionTitle/sourceFileName + documentId
```

如果 `citation` 字段已经包含完整引用，可直接使用。

## 检索结果为空时

如果没有结果：

```markdown
当前财报知识库中没有检索到与该问题直接相关的内容。可能原因包括：

1. 该公司的知识库尚未构建；
2. 相关财报未下载或未入库；
3. 查询条件过窄，例如年份或 form_type 限制不匹配。

可以先执行 `build_filing_kb` 或 `sync_filing_kb` 构建/同步该公司的知识库后再查询。
```

## 多条证据冲突时

如果检索结果存在冲突：

- 不要强行选择一个结论。
- 列出不同表述及来源。
- 说明需要人工核验。

示例：

```markdown
检索结果中存在两类解释：一类强调成本上升，另一类强调战略投入增加。由于当前片段没有明确量化各因素贡献，无法判断哪个因素占主导。
```

## 数值问题处理

知识库检索适合回答“原因”和“解释”，不适合精确数值。

如果用户问：

> 经营利润下降了多少？

应优先使用 `query_financial_metrics`。

如果用户问：

> 经营利润下降的原因是什么？

应使用 `retrieve_filing_kb`。

如果用户问：

> 经营利润下降了多少，原因是什么？

推荐步骤：

1. 调用 `query_financial_metrics` 获取数值变化。
2. 调用 `retrieve_filing_kb` 获取原因证据。
3. 合并回答：先数值，后原因。

## 禁止行为

- 不要说“财报显示”但没有引用 chunk。
- 不要把模型常识当成公司财报事实。
- 不要在没有检索结果时编造原因。
- 不要混淆不同年份、不同公司、不同表单的证据。
- 不要把 risk factor 中的风险当成已经发生的原因，除非检索片段明确说明该风险已影响当期业绩。
